# Kaboo Retro Game Color Reduction Tool 計画書

## 1. 目的

レトロゲームのハードウェア仕様に沿った減色画像を作成できる、独立動作型のWebツールを構築する。

本ツールは以下を満たす。

- 外部サービスや外部ライブラリに依存せず、ブラウザだけで動作する
- 各ゲーム機の色数、パレット、BG、スプライトの制約を考慮した減色を行う
- 減色画像だけでなく、使用したBGパレットとスプライト情報も同時に表示する
- ユーザーが指定した矩形範囲では、ディテール保持を優先した減色方針を適用できる
- マウスまたはタッチ操作だけで主要機能を完結できる
- スマートフォンでも破綻しないUIを提供する

## 2. 対象ハードウェア

対象機種は以下の5機種とする。

- ファミリーコンピュータ
- セガ・マスターシステム
- ゲームボーイ
- ゲームボーイカラー
- ゲームギア

## 3. 基本方針

### 3.1 独立動作

以下の構成を採用し、外部ツールや外部JavaScriptライブラリに依存しない。

- `index.html`
- `style.css`
- `main.js`
- 必要に応じて分割したローカルJavaScriptファイル

画像読み込み、減色処理、タイル解析、BGパレット生成、スプライト抽出、結果表示はすべてブラウザ内で完結させる。

操作系は次を前提にする。

- 主要操作はマウスだけで完結する
- タブレット / スマートフォンでは同等機能をタッチ操作で利用できる
- キーボードショートカットは補助機能として扱い、必須操作にしない

### 3.2 制約重視の処理順

単純な全体減色ではなく、ハードウェア制約を優先した次の順で処理する。

1. 入力画像を読み込む
2. 対象機種を選択する
3. 必要に応じてディテール優先範囲を矩形で指定する
4. タイル単位で画像を解析する
5. BGパレット候補を生成する
6. BGで表現できる領域を割り当てる
7. BGで収まらない領域をスプライト候補として抽出する
8. ROI内ではディテール保持を優先し、BG/スプライト配分とパレット割り当てを再評価する
9. BGとスプライトの制約を反映した最終減色画像を生成する
10. 減色画像、BGパレット、スプライト情報を同時に表示する

## 4. 対応すべき制約

機種ごとに以下の制約をモデル化する。

- 1色あたりの色深度
- 同時表示色数
- BGパレット数
- BGパレットあたりの色数
- スプライトパレット数
- スプライトパレットあたりの色数
- タイル単位でのパレット制約
- 透明色の有無と扱い
- スプライトサイズ候補
- 走査線ごとのスプライト表示制限

特に以下を重視する。

- BGパレット制限
- スプライト制限
- パレット制限
- ROI内のディテール優先制御

## 5. システム構成案

### 5.1 モジュール構成

実装は以下の責務分割を想定する。

#### `machine-profiles.js`

各機種のハードウェア制約を定義する。

#### `image-loader.js`

画像ファイルの読み込み、Canvasへの展開、ピクセルデータ取得を担当する。

#### `color-quantizer.js`

機種ごとの色深度に合わせた色丸め、最近傍色選択、必要に応じてディザリングを担当する。

#### `tile-analyzer.js`

画像をタイル単位に分割し、各タイルの使用色、色数、近似コストを算出する。

#### `roi-manager.js`

矩形で指定されたディテール優先範囲を管理し、マウス/タッチによる追加・移動・リサイズ、およびタイルやスプライト候補への重み付けを担当する。

#### `palette-solver.js`

BGパレット候補生成、タイルへのパレット割り当て、ROI重みを考慮した制約超過判定を担当する。

#### `sprite-extractor.js`

BGで表現しきれない領域をスプライト候補として抽出し、ROI内の候補を優先しながらスプライトパレット候補も生成する。

#### `renderer.js`

最終減色画像、BGパレット一覧、スプライト一覧、警告表示を描画する。

#### `main.js`

UIイベントと全体フロー制御を担当する。

#### `input-controller.js`

マウス操作とタッチ操作を統一的に扱い、ROI作成、選択、移動、リサイズ、画像パン/ズームなどの入力制御を担当する。

### 5.2 データモデル

主要な内部データは以下を想定する。

- `MachineProfile`
- `TileInfo`
- `RegionOfInterest`
- `PaletteSet`
- `BgAssignment`
- `SpriteCandidate`
- `ReductionResult`
- `InputState`

## 6. 画面構成案

### 6.1 上部設定エリア

- 画像読み込み
- クリップボード画像の貼り付け案内
- 対象機種選択
- ディザリングON/OFF
- BG優先 / スプライト優先の重み調整
- ディテール優先矩形の追加 / 編集 / 削除
- 透明色設定

画像入力は以下の2経路に対応する。

- ファイル選択による画像読み込み
- クリップボードからの画像貼り付け (`Ctrl+V`)

操作入力は以下に対応する。

- マウスドラッグによるROI矩形の追加
- マウス操作によるROIの移動 / リサイズ / 削除
- タッチドラッグによるROI矩形の追加
- タッチ操作によるROIの移動 / リサイズ / 削除

### 6.2 中央キャンバスエリア

- 最終減色画像
- BGのみ表示
- スプライトのみ表示
- ROIオーバーレイ表示
- 制約違反の強調表示

PC / スマートフォンともに、縦スクロール前提の単カラム構成とする。

### 6.3 下部情報エリア

- BGパレット一覧
- スプライトパレット一覧
- スプライト候補一覧
- ROIごとの優先度統計
- タイル数、使用色数、制約超過数などの統計

BGパレット、スプライト情報、ROI統計、警告は、下部の折りたたみセクションまたはタブに集約する。

## 7. 機種別実装方針

### 7.1 ゲームボーイ

4階調を持つ単純な構成のため、最初の完成対象とする。

### 7.2 ゲームボーイカラー

タイル単位パレット割り当てを重視し、ゲームボーイ実装の次に着手する。

### 7.3 セガ・マスターシステム

BGとスプライトのパレット制約を実装しやすいため、中盤で対応する。

### 7.4 ゲームギア

セガ・マスターシステム系の設計を流用しつつ、色表現差分を追加する。

### 7.5 ファミリーコンピュータ

属性単位やパレット制約の扱いが難しいため、最後に実装する。必要に応じて以下の2モードを検討する。

- 実機寄り厳密モード
- 実用寄り簡略モード

## 9. 優先順位

優先順位は以下とする。

1. 独立動作する最小Webアプリを成立させる
2. ゲームボーイで制約つき減色を完成させる
3. BGパレット表示を実装する
4. スプライト候補表示を実装する
5. ROIによるディテール優先制御を実装する
6. マウス / タッチのみで完結する操作系を整える
7. スマートフォン対応を整える
8. 他機種へ横展開する
9. ファミコンの厳密度を上げる

## 10. 想定リスク

- 機種ごとの実機制約の解釈差により、厳密再現が難しくなる可能性がある
- BGとスプライトの自動分離は画像によって最適解が揺れる
- ROIを複数指定した場合、領域同士の競合で期待どおりの優先付けにならない可能性がある
- マウス操作とタッチ操作でROI編集挙動がずれる可能性がある
- スマートフォン画面では情報量が多く、ROI編集と結果確認の両立が難しくなる可能性がある
- ファミコンは属性制約のため、他機種より実装難易度が高い
- 外部ライブラリなしのため、高度なクラスタリングや高速化は自前実装が必要
