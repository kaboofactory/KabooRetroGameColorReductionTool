# ファミコン完全実装計画

## 1. 目的

本計画書は、現在のWebアプリを「ファミリーコンピュータの制約に従って、入力画像をBGとスプライトへ分解しながら減色できるツール」へ段階的に完成させるための実装計画をまとめたものである。

目標は次のとおりとする。

- ファミコン専用ツールとして通しで使えること
- 画像入力から結果確認まで、ブラウザだけで完結すること
- BGパレット、属性領域、スプライト制約を反映した結果を確認できること
- ROI 1箇所をマウスドラッグで指定し、その範囲のディテール優先度を上げられること
- PC/スマートフォンともに単カラムUIで操作できること

## 2. 実装対象

初期リリースでは、対象機種をファミリーコンピュータのみに限定する。

実装対象は以下とする。

- 入力画像の読み込み
- クリップボード貼り付けによる画像取り込み
- ROI 1箇所の指定、移動、リサイズ
- BG用減色
- BGパレット生成
- 属性領域ごとのBGパレット割り当て
- スプライト候補抽出
- スプライトパレット生成
- BG / Sprite 分離プレビュー
- 制約超過警告の表示

## 3. 完全実装の定義

このプロジェクトでいう「ファミコン完全実装」は、少なくとも以下を満たす状態を指す。

### 3.1 BG 側

- 画面を `8x8` タイル、`16x16` 属性領域として扱う
- universal background color を1色決める
- BGサブパレットを4組生成する
- 各属性領域に4組のうち1組を割り当てる
- 各属性領域内の全ピクセルが、割り当てパレットで表現できるか評価できる
- 表現しきれない色や誤差を可視化できる

### 3.2 Sprite 側

- BGで表現しきれないピクセルをスプライト候補として抽出する
- 候補を `8x8` または `8x16` 単位へまとめられる
- スプライトパレット4組を生成できる
- 各スプライト候補にパレットを割り当てられる
- 走査線ごとの8枚制限を警告できる

### 3.3 ROI 側

- ROIは1箇所だけ持つ
- ROIは画像上でマウスドラッグして作成できる
- ROIはマウスまたはタッチで移動・リサイズできる
- ROI内では、BG割り当て誤差やSprite化優先度の重みを上げられる

### 3.4 表示側

- 最終画像
- BGのみ
- Spriteのみ
- 元画像
- BGパレット一覧
- Spriteパレット一覧
- 属性領域の割り当て状況
- スプライト候補一覧
- 警告一覧

を同一画面で確認できるようにする。

## 5. 実装方針

### 5.1 処理の基本順序

処理は以下の順で組み立てる。

1. 入力画像をファミコン64色へ量子化する
2. universal background color を決める
3. 属性領域ごとの色分布を集計する
4. BGサブパレット4組を決める
5. 各属性領域へBGサブパレットを割り当てる
6. スプライトを使う `8x8` 領域を先に決める
7. スプライト使用領域ではBGを評価対象から外し、残り領域でBG品質を最大化する
8. スプライト候補からスプライトパレット4組を決める
9. 候補を `8x8` / `8x16` スプライトへまとめる
10. BG / Sprite の合成結果と警告を出す

### 5.2 制約駆動の段階的収束

今後の中核方針は、「最初に強く統合してから戻す」のではなく、「できるだけ元絵に忠実な状態から始め、必要になった制約だけ段階的に締める」方式とする。

この方針では、以下を原則とする。

- 初期状態では BG タイル統合、Sprite タイル統合、ROI 外 Sprite 削減を必要最小限に留める
- 制約違反がなければ、その時点の結果を採用する
- 制約違反がある場合だけ、違反している種類の制約に対応する締め付けを1段階だけ強める
- 再評価を繰り返し、全制約が満たされた時点で停止する
- 停止時に BG / Sprite / パレット資源が自然に余るのは許容するが、品質を落としすぎる早期統合は避ける

これは、余剰資源を後から再分配するのではなく、最初から「締める必要が出るまで締めない」探索へ切り替えることを意味する。

### 5.3 締め付け対象の優先順位

制約違反が起きた場合、すべてを一度に変えるのではなく、違反内容に応じて局所的に締める。

- BGユニーク `8x8` タイル数超過
  - 類似BGタイル統合のしきい値を段階的に上げる
  - ROI外タイルを先に統合対象とする
  - 反転で再利用可能な同形タイルがあれば優先的にまとめる

- Spriteユニーク `8x8` タイル数超過
  - 類似Spriteタイル統合のしきい値を段階的に上げる
  - ROI外の低優先Spriteから先に代表タイルへ寄せる
  - 反転再利用を最大限使う

- 属性セル不足
  - BGサブパレット候補の代表色寄せを段階的に強める
  - ROI外で BG 表現へ寄せる圧力を強める
  - それでも不足する場合に限り Sprite 逃がしを増やす

- 走査線ごとの8枚Sprite制限超過
  - ROI外・低誤差・低密度のSprite候補から段階的に除外する
  - 隣接候補の統合を強める
  - それでも難しい場合は BG 側の近似を受け入れる

- BGパレット未活用・偏り過多
  - 使用率の低いBGパレットを検出する
  - そのBGを既存パレットの複製ではなく、取りこぼし色群の代表へ差し替える
  - 属性セル割り当てを再計算し、全体誤差が改善するか比較する

### 5.4 段階的収束ループ

最終的な探索ループは以下を想定する。

1. 入力画像を量子化し、初期BG / 初期Sprite候補を作る
2. できるだけ統合を抑えた状態で一度結果を生成する
3. 制約違反を検出する
4. 違反している制約のうち、最も支配的なものを特定する
5. 対応する締め付けパラメータを1段階だけ強める
6. 再生成して比較する
7. 制約がすべて満たされるまで繰り返す
8. 収束後の結果を最終出力とする

ここで重要なのは、次の2点である。

- 「余った枠があるから後で戻す」処理は原則行わない
- 代わりに、「必要になるまで締めない」ことで結果的にリソース利用率を高める

ただしBGパレットについては例外的に、明らかに未使用または低使用の枠がある場合だけ、その枠を「余ったまま」にせず再活用候補として再シードする。
これは後戻りというより、未活用資源の明示利用とみなす。

### 5.5 ROI の効かせ方

ROIは「色数を直接増やす」機能ではなく、以下の重みづけとして使う。

- universal background color 決定時の重み
- BGサブパレット候補選定時の重み
- 属性領域へのパレット割り当て誤差の重み
- BGに載せずSpriteへ逃がす優先度
- スプライト候補統合時の優先度

段階的収束方式へ移行したあとも、ROI は「最後まで締めにくい領域」として働くようにする。

- ROI内は類似タイル統合の対象から最後に回す
- ROI内のSprite候補は落選・統合の優先順位を下げる
- ROI外から先に制約解消を進める

### 5.6 Sprite優先領域を先に切る方針

今回の方針では、Spriteは「BGの上に追加で少し描く補修」ではなく、「その `8x8` はSprite担当にして、BGはそこを無理に背負わない」ための資源として扱う。

つまり、Spriteを使う領域では次を原則とする。

- その `8x8` 領域の主たるディテールはSpriteが担当する
- BGはその領域で無理に元絵再現を目指さない
- BGの色数・パレット・タイル資源は、Spriteを使わない領域へ優先配分する
- Sprite透明画素から見えるBGは、細部再現ではなく下地や面のつながりを担う

この方式では、最初に「どの `8x8` をSprite優先領域にするか」を決め、その後でBGを再最適化することが重要である。

処理順は次を基本とする。

1. 各 `8x8` タイルについて、BG単独での破綻量を評価する
2. Spriteへ切り出した方が得なタイルを選ぶ
3. 選ばれたタイルは BG品質評価から除外、または重みを大きく下げる
4. 残りタイルだけで BGパレットと属性割り当てを再最適化する
5. 最後に Sprite を合成して全体結果を出す

この考え方により、「Spriteを増やしたのにBGも同じ場所を頑張ってしまい、全体が共倒れになる」状態を避ける。

### 5.7 低使用BGの再活用方針

カラフルな画像であっても、初期のBGパレット生成と属性セル割り当ての結果次第では、`BG0〜BG3` の一部がほとんど使われないことがある。
この状態は、実機制約上は問題なくても、見た目の色域を活かし切れていない可能性が高い。

そこで今後は、次の再活用ループを導入する。

1. 各BGパレットの使用率を、属性セル数または利用ドット数で計測する
2. 閾値未満の低使用BGパレットを検出する
3. 複数の低使用BGパレットがある場合は、それぞれへ異なる未カバー色群を割り当てる
4. 属性セル割り当てを再計算する
5. まだ未使用BGが残る場合は、数回だけ再配置を繰り返す
6. 全体誤差、未解決画素、BG偏り、Sprite依存度を比較し、改善した場合だけ採用する

この再活用は、次の条件を満たす場合にのみ有効化する。

- 低使用BGパレットが存在する
- その再シードで全体誤差が改善する
- 走査線制限やSprite依存が悪化しすぎない

これにより、「本来もっとカラフルにできるのに、BGの一部が遊んでいる」状態を減らす。

### 5.8 UI 方針

- PC版もモバイル版も1カラムとする
- 画像キャンバスを中心に、設定と結果を縦に並べる
- 高度な設定は増やしすぎず、変換に必要なものへ絞る
- ディザリングは初期リリース対象外とする

## 8. 既知の難所

- BGサブパレット4組の最適化は全探索が重くなりやすい
- ROI優先と全体最適は衝突しやすい
- Sprite化を増やすと走査線制限へ当たりやすい
- Sprite優先にしたタイルの下地BGをどう簡略化するかが難しい
- Sprite優先タイルを増やしすぎると、走査線とSprite枚数が急増しやすい
- 低使用BGを再シードしても、局所的には改善するが全体割り当てでは悪化する場合がある
- 「未使用BGを埋めること」自体が目的化すると、かえって見た目が悪化する危険がある
- 複数の低使用BGへ似た色群を再配置すると、結局片方しか使われないことがある
- 複数制約を同時に締めると、別の制約を悪化させる相互作用がある
- 段階的収束の停止条件が弱いと、反復回数が増えやすい
- 実機完全再現を狙いすぎると、ツールとしての分かりやすさが落ちる可能性がある

このため初期実装では、次の方針を採る。

- まずは「必要になるまで締めない」探索を優先する
- 実機制約に触れる箇所は警告と可視化を厚くする
- 完全自動化が難しい部分は、内部スコアリングと段階的収束ループを改善しながら精度を上げる
